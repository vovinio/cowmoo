1 · The principle

The diagnosis: PM is leaking spec-grade prose into chat. The user reads paragraphs to verify what PM understood — slow, error-prone, easy to nod past mistakes. Fix: separate the two surfaces.

Spec file — the artifact

  • Elaborate
  • Workflows, examples, edge cases
  • Reasoning preserved verbatim
  • Read once, inherited forever (planner + builder)
  • Thoroughness pays

Chat — the steering wheel

  • Dense-but-concrete
  • Compressed representations of same content
  • Pickers for forks, diffs for changes
  • Scanned in 5 seconds
  • Brevity that preserves detail pays

The misunderstanding test: could a wrong interpretation of what the user said look identical to the right one in this rendering? If yes, the compression dropped load-bearing detail — re-add it as a named decision or a worked example, not as a longer paragraph.

2 · Compression forms

The techniques that pack context into something the eye can scan. Pick the form that survives the misunderstanding test.

FormExamples
Worked example
over abstract description		 NO "Autocomplete suggests matches from existing accounts" YES User types acme → dropdown shows 3 matches → picks one → form fills
Named decisions
over narration		 NO "We'll use magic-link auth with 30-day sessions and explicit logout" YES Auth: magic link · Session: 30d · Logout: explicit
Diff
over change-narration		 NO "We're changing it so that retries now happen 3 times" YES Before: 1 retry → After: 3 retries
Mini-flow
over paragraph (≤8 words/step)		 YES 1. User submits invalid email   2. Inline error appears under field   3. Submit button stays disabled
Picker options
with consequence in description		 NO "Magic link auth" YES "Magic link (no password reset flow; requires working email)"

Fix 3 · Output-style "Conversation Focus" reframing

herd/pm/.claude/output-styles/product-specification.md:14-27
Current — opens with 5 open-ended questions
Ask concrete questions:
- "Can you give me a specific example?"
- "What exactly happens when that fails?"
- "Who can do this — everyone or specific roles?"
- "What if there's no data yet?"
- "What happens if they stop midway?"
Proposed — proposal-first, open-question fallback
Default (have a prior):
  "If validation fails I'd default to:
   inline error · preserve values · don't reset.
   Or: full-page error / toast + reset?"
  [picker]

Fallback (no prior, genuinely don't know):
  "What's the user's role here, and what
   authority do they have in this flow?"

Why: the open-ended questions contradict CLAUDE.md:11 ("Don't ask — propose"). The user's job is to react to options, not invent solutions. Open questions are the fallback when PM genuinely has no prior to seed options from.

4 · Two structural additions

A · Name the two-surface principle in CLAUDE.md

herd/pm/CLAUDE.md — new item under "How You Work"

Add a short section between current items 3 and 4:

**Two surfaces, different jobs.**
- **Spec file** (artifact) — elaborate. Workflows, examples,
  edge cases, reasoning. This is what planner/builder
  inherit; thoroughness pays.
- **Chat** (steering wheel) — dense-but-concrete
  representations of the same content. Designed to be
  scanned in 5 seconds, not read.

Never paste spec-grade prose back into chat for the user
to verify. The user verifies via compressed forms (named
decisions, diffs, worked examples, pickers). Spec file is
the long version; chat is the short version.

B · The misunderstanding test in output-style

herd/pm/.claude/output-styles/product-specification.md — new mini-section

Add to ## Tone or as a new ## Before You Send:

Before sending any echo / summary / confirmation, ask:
*could a wrong interpretation of what the user said look
identical to the right one in this rendering?*

If yes, add the load-bearing detail back — usually as a
named decision or a worked example, not as a longer
paragraph.

5 · What's NOT changing

  • Templates (domain.md, entity.md, feature.md, product.md) — already elaborate, the right place for richness. Spec artifact is good as-is.
  • Picker rule in CLAUDE.md — already correctly enumerated. /ideate and /review already apply it well.
  • Working-notes pipeline — Rule 1/2/3 are sound. Notes → digest → spec is the right flow.
  • /digest batching — write → re-read → verify per domain is correct. Fix is in what chat shows, not when it shows it.
  • /draft "Where we left off" paragraph — lives in WORKING-NOTES.md (artifact, not chat); a short narrative is the right form for resumption context.
  • /review per-finding presentation — already uses pickers and structured quote/wrong/options/recommendation blocks.
  • No new rule file — all additions belong in output-style + CLAUDE.md + three skill bodies. None is "always-needed canonical content" in the rule sense.
